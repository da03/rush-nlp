What is a “program”?

Every decision in the pipeline is a PAW program: a short natural-language spec compiled into a tiny, specialized model.

→
finetune compilerpaw-ft-bs48
→

→

  • Cheap + specialized: each program does one narrow job (route, classify, answer, judge).
  • Rule + fuzzy: deterministic Python (links, facts, redundancy) wraps the fuzzy PAW calls.
  • Reproducible: IDs are pinned & committed — the server runs exactly what was tested.

Page-aware routing

The same question routes differently depending on the page — the domain_router sees the page context.

Default to the current page’s domain; escape only when the subject clearly belongs elsewhere.

A flat answerer is the fallback; a topic router sends detailed questions to specialized sub-answerers, each fed only its own slice of facts.

Resource routers (rule + fuzzy)

When a classifier emits a generic label, a fuzzy selector picks specific deep links from a rule-supplied candidate list.

slide_selector

“reinforcement learning slides” → the actual lecture deck(s), not just the syllabus.

code_selector

“the python sdk” → Python repo; “npm package” → JS repo; bare “source code” → the org page.

Python renders the candidates & maps the selector’s output to links; the PAW program only does the fuzzy matching.

The course Piazza RAG branch

On the course page only, a parallel branch answers from endorsed-public Piazza threads — over-call → rerank → answer → merge.

BM25 searchrecall (cheap)
→
piazza_selectorwhich threads
→
piazza_answerersynthesize / decline
→
piazza_mergemain · augment · branch
  • No brittle gate: always search; the question-aware merge judge (sees both answers) decides whether Piazza wins, augments, or stays out.
  • Faithful: if the answerer can’t answer from the threads, it declines — and the branch shows nothing (no misleading citation).
  • Concurrent & private: runs alongside the main answer; only public, instructor-answered threads are indexed.

Facts: injected & date-aware

Domains inject facts at inference time — editing facts needs no recompile (only specs do).

  • Runtime injection: the right slice of YAML/Markdown is selected per query (the RAG seam).
  • Computed in Python (course): today’s date (America/Toronto), next deadline across chats & assignments, and date-aware status — “Assignment 3 is not yet released; releases July 9.”
  • Hot-reload: the Piazza index re-syncs hourly and reloads within 5 min, no restart.

So “when is the next deadline” and “is A3 out yet” are correct on any given day without recompiling a single program.

Evaluation gates every change

The real lesson: per-component metrics miss outcome regressions. Each domain has an outcome suite that grades the final user-facing result.

Outcome suites (the deploy gate)

  • course_e2e · site_e2e
  • neuralos_e2e · pawsite_e2e
  • graded on: which domain, result kind, content, must-decline.

The discipline

  • Every reported bug becomes a test before the fix.
  • Mine real per-origin traffic; eyeball, don’t trust numbers.
  • Run on the same backend the server uses.

All four domains: 100% on the live backend.

Running it: backends & resilience

  • Two interchangeable backends: local_sdk (runs the tiny models in-process, no rate limit) and remote_infer (the PAW /api/v1/infer API, concurrent).
  • Auth matters: remote_infer needs a valid PAW_API_KEY (X-API-Key). A missing/stale key → anonymous rate limit → blank answers — a gotcha we hit and fixed.
  • Resilience: retry-on-empty for transient API blips; the index hot-reloads; pinned IDs mean reproducible serving.
  • One widget contract: render answers, links, related citations, feedback — so frontends don’t drift.
