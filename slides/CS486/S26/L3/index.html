Uninformed search

  • Treats every state the same
  • No idea which is closer to goal

Heuristic search

  • Estimates distance to goal
  • Expands the most promising state

Two quantities at every node \(n\)

\(g(n)\) — cost so far

Actual cost of the path from start to \(n\).

\(h(n)\) — heuristic

Estimated cost from \(n\) to a goal.

Combined: \(f(n) = g(n) + h(n)\) — total estimated cost through \(n\).

What makes a good heuristic?

  • Problem-specific
  • Non-negative
  • \(h(\text{goal}) = 0\)
  • Easy to compute (no search)

Lowest-cost-first search (LCFS)

  • Frontier = priority queue, key = \(g(n)\)
  • Pop the cheapest path so far

a.k.a. Uniform-cost search (UCS) in RN, or Dijkstra's algorithm in algorithms textbooks.

A* search

  • Frontier = priority queue, key = \(f(n) = g(n) + h(n)\)
  • Pop the path with smallest total estimated cost

A* combines LCFS's cost-so-far with GBFS's goal-pull.

Trace: A*

Path notation SBE: 6 = path with \(f = g+h = 6\). g in blue, h in green.

Frontier: (S: 8) pop S → (SC: 4, SB: 8) pop SC → (SB: 8, SCH: 102) pop SB → (SBE: 6, SBD: 11, SCH: 102) pop SBE → (SBD: 11, SBEF: 17, SCH: 102) pop SBD → (SBDG: 11, SBDF: 17, SBEF: 17, SCH: 102) pop SBDG — goal! cost = 11

A* — properties

Space\(O(b^m)\) worst case; often much better
Time\(O(b^m)\) worst case; often much better
Complete?Yes under the usual finite-branching condition
Optimal?Yes if \(h\) is admissible

A* is optimal when \(h\) is admissible

Admissible: \(h(n) \le\) true cost from \(n\) to any goal, for every \(n\). (Never over-estimates.)
Theorem. If \(h\) is admissible, A* returns an optimal path.
Proof sketch (contradiction):
  1. Suppose A* returns a path with cost \(C^*\), but the true optimal cost is \(C' < C^*\).
  2. The optimal path passes through some node \(N\) still on the frontier when A* terminated. By admissibility, \(f(N) = g(N) + h(N) \le C'\).
  3. So \(f(N) \le C' < C^*\). But then A* would have popped \(N\) before the returned path. Contradiction.

A* is optimally efficient

No other optimal algorithm using the same heuristic \(h\) expands fewer nodes than A*.

Intuition: any algorithm that doesn't expand all nodes with \(f(n) < C^*\) risks missing the optimum.

Manhattan distance

Sum of grid distances of each tile from its goal position.

Misplaced tile count

Number of tiles not in their goal position.

Both are admissible.

Recipe: relax the problem

  1. Relax the problem by dropping constraints.
  2. Solve the relaxed problem (without search).
  3. The cost of that solution is an admissible heuristic for the original.
8-puzzle constraints: a tile moves from A to B if (a) A and B are adjacent and (b) B is empty.

Dropping a constraint gives an easier problem, and its optimal cost lower-bounds the original.

Quick check: heuristic from relaxation

Q. Drop the "B is empty" constraint: a tile can move from A to B if A and B are adjacent. Which heuristic does this give?

  1. Manhattan distance
  2. Misplaced tile count
  3. Another heuristic not listed
A — Manhattan distance. Without the "B must be empty" constraint, each tile slides freely along the grid to its goal — cost = number of grid steps = Manhattan distance.

Dominating heuristics

Definition. \(h_2\) dominates \(h_1\) if \(h_2(n) \ge h_1(n)\) for all \(n\), and \(h_2(n) > h_1(n)\) for some \(n\).
Theorem. If \(h_2\) dominates \(h_1\), A* with \(h_2\) expands no more nodes than A* with \(h_1\). (Higher = better, as long as still admissible.)

Q. Manhattan distance vs. Misplaced tile — which dominates?

Manhattan dominates. Each misplaced tile contributes at least 1 to both heuristics, but Manhattan adds the actual grid distance — so Manhattan \(\ge\) Misplaced, strictly > whenever any tile is more than one step from its goal.

Cycle pruning

Don't follow a path that revisits a node already on the current path.

// In the generic search algorithm, when expanding path <n0, ..., nk>:
for each neighbour n of nk:
    if n ∉ <n0, …, nk>:        # skip cycles
        add <n0, …, nk, n> to frontier

Time check is linear in path length. Saves DFS from infinite loops.

Multi-path pruning

Don't expand a node we've already expanded. Cycle pruning is a special case.

explored ← { }
while frontier is not empty:
    pop path <n0, …, nk>
    if nk ∉ explored:           # multi-path pruning
        add nk to explored
        if goal(nk): return path
        for each neighbour n of nk:
            add <n0, …, nk, n> to frontier

Caveats of multi-path pruning

  • Paths to already-explored nodes still get pushed — just skipped when popped
  • Saves time, spends memory (we keep the explored set)
  • And… may sacrifice optimality (see next slides)

Does multi-path pruning preserve optimality?

Q1. LCFS with multi-path pruning — can it discard the optimal solution?

No. LCFS always pops the cheapest path first — so the first time a node is popped, the path through it is optimal. Pruning later paths to the same node is safe.

Q2. A* with admissible heuristic + multi-path pruning — can it discard the optimal?

Yes — it can. A* pops by \(f = g + h\), not pure \(g\). A node's first pop might use a suboptimal \(g\). The next slide shows an explicit counterexample.

A* + multi-path: counterexample

Optimal path is S→C→E→G (cost 23). With multi-path pruning, A* returns S→B→E→G (cost 25) instead.

When does multi-path pruning fail?

Setup. Node \(n\) is explored first via parent \(p\) (cheaper \(f\)). Later a path through \(p'\) reaches \(n\) with smaller \(g\) (cheaper actual cost) but larger \(f\) (so it was popped later).
  1. By the order of pops: \(f(\langle\ldots,p,n\rangle) \le f(\langle\ldots,p',n\rangle)\).
  2. So \(g(p) + \text{cost}(p,n) + h(n) \le g(p') + \text{cost}(p',n) + h(n)\).
  3. Simplify: \(g(p) + \text{cost}(p,n) \le g(p') + \text{cost}(p',n)\).   (*)

For the path-via-\(p'\) to actually be better, we need the reverse: \(g(p') + \text{cost}(p',n) < g(p) + \text{cost}(p,n)\).   (**)

Both (*) and (**) can hold only if A* mis-ranked the \(f\)-values — which happens when the heuristic violates:

\(h(p) - h(n) \le \text{cost}(p,n)\)

i.e., the heuristic drops too fast across an edge. Fix that, and multi-path pruning is safe.

Consistent (monotone) heuristic

Definition. \(h\) is consistent if for every edge \(m \to n\),

\(h(m) \le \text{cost}(m, n) + h(n)\).

Equivalently: \(h(m) - h(n) \le \text{cost}(m, n)\). The heuristic decreases by no more than the edge cost.
Theorem. If \(h\) is consistent, A* with multi-path pruning is optimal.

Most natural heuristics (Manhattan, straight-line distance, …) are consistent — consistency is the stronger and friendlier condition.
