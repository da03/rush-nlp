Why search?

  • Problems that require sequential decision making: a series of moves, not a single closed-form computation.
  • Used when no efficient direct algorithm exists. (Unlike, e.g., solving a linear system: that has fast specialized solvers.)
  • Often easy to verify a candidate solution, but hard to find one — the defining feature of NP-hard problems.
  • A search algorithm explores the space of possibilities systematically, instead of trial-and-error.

What is a search problem?

A search problem is defined by:

  • A set of states.
  • An initial state.
  • Goal states — or a goal test: a boolean function on a state.
  • A successor (neighbour) function: which states can we reach from this one?
  • (Optional) a cost on each action.

A solution is a path from the initial state to a goal state — optionally one with minimum total cost.

Graph → search tree

  • Each state is a node; each action is a directed edge.
  • Search is: find a path from S (start) to G (goal).
  • We rarely build the graph upfront — it can be huge or infinite. Instead, grow a search tree from S, expanding one path at a time.
  • The frontier is the set of paths that are candidates for expansion.

For this lecture, every edge has cost 1.

Choosing among multiple formulations

  • The state definition determines the nodes; the successor function determines the edges.
  • We usually want to minimize the number of nodes and edges.
  • The state definition also makes the successor function easier or harder to implement.
Alternative formulation: a state is 9 coordinates, \((x_i, y_i)\) for each tile \(i\), \(0 \le i \le 8\).
Same problem, ~2× the space; successor function now “swap the empty tile's coords with a neighbour's coords”.

Generic search algorithm

Maintain a frontier of paths from \(s\). At each step, pull one out, check the goal, and add neighbouring extensions.

// Inputs: graph, start node s, goal test goal(n)
function Search(graph, s, goal):
    frontier ← { <s> }
    while frontier is not empty:
        select and remove a path <n0, …, nk> from frontier
        if goal(nk):
            return <n0, …, nk>
        for each neighbour n of nk:
            add <n0, …, nk, n> to frontier
    return “no solution”

Every uninformed search algorithm in this lecture is this skeleton; they only differ in how they pick the next path out of the frontier.

Depth-first search (DFS)

  • Treat the frontier as a stack (LIFO).
  • Expand the last path added.
  • Goes down one path to the bottom before backtracking to alternatives.
Push neighbours of the last node, pop the newest. Same generic skeleton; the choice of data structure is the entire difference.

Trace: DFS on a search graph

Push neighbours in alphabetical order. Step numbers = order of expansion.

Frontier: (S) (D, E, P) (D, E, Q) (D, E) (D, H, R) (D, H, F) (D, H, C, G) goal G reached!

DFS — properties

Let \(b\) = branching factor, \(m\) = max depth of the search tree, \(d\) = depth of the shallowest goal.

Space complexity\(O(bm)\) — linear in \(m\). Remembers the current path (\(m\) nodes) + up to \(b\) siblings per level.
Time complexity\(O(b^m)\) — exponential in \(m\). May visit the entire tree.
Complete?No. Can get stuck in an infinite path (cycle or unbounded depth).
Optimal?No. Returns the first goal it finds, regardless of cost.

DFS — failure modes

Cycle → not complete

Loop \(H\to I\to J\to H\) traps DFS forever; goal \(G\) is never reached.

Returns first goal → not optimal

DFS finds the long \(S\to E\to J\to F\to G\) path; the short \(S\to D \dashrightarrow G\) is ignored.

When to use DFS

Good fit

  • Memory is restricted.
  • Many solutions exist (so getting some goal quickly is OK).
  • Solutions are deep / paths are long.

Bad fit

  • There are infinite paths or cycles (without cycle pruning).
  • Solutions are shallow.
  • You need the optimal (shortest) solution.

Breadth-first search (BFS)

  • Treat the frontier as a queue (FIFO).
  • Expand the first path added.
  • Explore the tree level-by-level — the path with the fewest arcs first.
The only difference from DFS is the data structure: queue instead of stack.

Trace: BFS on a search graph

Push neighbours in alphabetical order. The graph contains multiple paths to P and B.

Frontier: (S) (D, E, P) (B, C, H, J, Q) (P, B, G) G is the goal — found!

BFS — properties

Let \(b\) = branching factor, \(d\) = depth of the shallowest goal.

Space complexity\(O(b^d)\) — exponential in \(d\). Must hold the whole frontier at level \(d\).
Time complexity\(O(b^d)\) — exponential in \(d\). Visits the entire top \(d\) levels in the worst case.
Complete?Yes. Explores level by level until a goal is found.
Optimal?No in general, but guaranteed to find the shallowest goal — which is optimal when all edges have equal cost.

When to use BFS

Good fit

  • Memory is plentiful.
  • You want the solution with the fewest arcs.
  • Cycles are possible.

Bad fit

  • All the solutions are deep.
  • The branching factor is large.
  • The graph is large or generated on the fly.

BFS vs DFS — Q1

Q. Memory is very limited. Which would you choose?

  1. BFS
  2. DFS
  3. Both are fine
  4. Neither
Answer: B — DFS. DFS uses linear memory \(O(bm)\); BFS uses exponential \(O(b^d)\).

BFS vs DFS — Q2

Q. All solutions are deep in the search tree. Which would you choose?

  1. BFS
  2. DFS
  3. Both are fine
  4. Neither
Answer: B — DFS. DFS dives down long paths quickly; BFS would waste time enumerating every shallower level first.

BFS vs DFS — Q3

Q. The graph contains cycles. Which would you choose?

  1. BFS
  2. DFS
  3. Both are fine
  4. Neither
Answer: A — BFS. DFS can get stuck looping forever; BFS is unaffected (it always advances by level).

BFS vs DFS — Q4

Q. The branching factor is huge (or infinite). Which would you choose?

  1. BFS
  2. DFS
  3. Both are fine
  4. Neither
Answer: B — DFS. BFS's frontier blows up with \(b\); DFS keeps memory linear regardless of \(b\).

BFS vs DFS — Q5

Q. You must find the shallowest goal. Which would you choose?

  1. BFS
  2. DFS
  3. Both are fine
  4. Neither
Answer: A — BFS. BFS is guaranteed to find the shallowest goal; DFS may return a deep one first.

BFS vs DFS — Q6

Q. All solutions are very shallow. Which would you choose?

  1. BFS
  2. DFS
  3. Both are fine
  4. Neither
Answer: A — BFS. BFS finds the shallowest goal first; DFS may dive into a deep dead-end before checking shallower neighbours.

Combining the best of BFS and DFS

BFS DFS
Space\(O(b^d)\) — exponential\(O(bm)\) — linear
CompletenessYesNo — can loop forever

Can we have both linear space and completeness?

Iterative-Deepening Search (IDS): for each depth limit \(\ell = 1, 2, 3, \ldots\), run depth-first search up to depth \(\ell\). Stop when you find a goal.

Trace: IDS on a search graph

Same graph as BFS. Within each depth limit, push children in alphabetical order; pop LIFO (DFS).

Depth limit: 1 — visit S only. 2 — visit S, P, E, D (DFS, alphabetical push order). 3 — explore level 3, no goal yet. 4 — S → P → Q → G. Goal found!

IDS — properties

Space complexity\(O(bd)\) — linear in \(d\). Each iteration is a depth-bounded DFS.
Time complexity\(O(b^d)\) — exponential in \(d\). Repeats work but dominated by the last iteration.
Complete?Yes. Always terminates at depth \(d\).
Optimal?No in general, but guaranteed to find the shallowest goal.

IDS pays a small constant-factor time overhead for the re-work, but gets BFS-like completeness with DFS-like memory. Best of both.

Summary: DFS vs BFS vs IDS

DFS BFS IDS
Frontier Stack (LIFO) Queue (FIFO) Stack, repeated at deeper limits
Space \(O(bm)\) \(O(b^d)\) \(O(bd)\)
Time \(O(b^m)\) \(O(b^d)\) \(O(b^d)\)
Complete? No Yes Yes
Optimal? No Shallowest goal Shallowest goal

\(b\) = branching factor; \(m\) = max depth of the search tree; \(d\) = depth of the shallowest goal.
